# Ethics in Technology

**CSCI 101 – Semester Writing Project**

## Overview

Strong written communication is essential in computer science, not just for code documentation, but for technical proposals and team collaboration. This semester-long project will strengthen your writing skills while you investigate a real-world issue: the ethical challenges of modern technology. Your final product is an argumentative research paper: you will take a clear position on an ethical question and defend it with evidence. You'll conduct research, draft and revise your work, and defend your findings in an oral examination during the final exam period. The project builds progressively through six stages, culminating in an oral examination, with submissions via GitHub Pull Requests, the same workflow used by professional developers worldwide.

- **6 graded checkpoints + an oral defense**, spread from Week 5 through finals week
- **All checkpoints due Sundays at 11:59 PM** (new stages typically open the following Monday)
- **Submissions via GitHub Pull Requests**, learn version control with real work
- **Final Paper:** ~2000 words (6–8 pages, not counting References page and title page)
- **Oral Examination:** 5–7 minute (max) presentation + Q&A during finals week
- **All work completed outside of class** (with feedback opportunities)

## Success Tips

- Start early and manage your time; checkpoints are roughly 2–3 weeks apart, but each builds on the last
- Use support resources: the Writing Center and research librarians
- Attend office hours for personalized guidance
- Commit often to track your revision history
- Each stage builds on previous work; don't fall behind

**Section 3: Annotated Bibliography**

**Acceptable:**

- Peer-reviewed academic articles

**Not Acceptable:**

- Wikipedia, personal blogs, or unverified websites

- Opinion pieces without empirical support

## Git Quick Reference

You work entirely inside **your own** project repository (created from the `CSCI101_F26Project` template). No fork, no `upstream` remote. For the full walkthrough, see the [Submission Recipe](SUBMISSION-RECIPE.md). Replace `#` with the stage number.

### For Each Stage:

    # 1. Sync your local copy
    git checkout main
    git pull

    # 2. Create a branch for this stage
    git checkout -b stage#

    # 3. Put your files in a stage# folder, then stage and commit
    git add stage#
    git commit -m "Stage # - FirstName LastName"

    # 4. Push
    git push -u origin stage#

    # 5. On GitHub: open a Pull Request (base: main, compare: stage#)

### Updating a Submission:

    # Make changes to your files, then:
    git add stage#
    git commit -m "Update Stage # - description of changes"
    git push
    # The Pull Request updates automatically!

## Appendix: APA 7th Edition Citation Examples

For the full reference with additional examples and formatting tips, see the [APA 7th Edition Citation Guide](APA_Citation_Guide.md). The examples below cover the most common source types in computer science projects.

### Why and What to Cite

Citations serve several essential purposes in your research paper. They give credit to the original authors whose ideas, research, or code informed your work; this is both an ethical obligation and a professional standard. Citations also strengthen your arguments by showing that your technical decisions are grounded in evidence from credible sources rather than personal opinion. Additionally, they allow readers to verify your claims and explore topics further.

You should cite sources whenever you:

- **Make technical comparisons:** When claiming one technology is faster, more secure, or better suited than another, cite benchmarks, documentation, or studies that support this.
- **Use concepts or techniques from others:** If you learned an algorithm, design pattern, or approach from a book, article, or tutorial, cite it.
- **Quote or paraphrase:** Any direct quotes require citations, as do paraphrased ideas that are not common knowledge.
- **Reference official documentation:** When explaining how a framework or tool works based on its documentation, cite that documentation. Official documentation is a valid and often preferred source for technology-specific information.
- **Build on existing research:** If your project extends or responds to prior work, cite that work.
- **Use code, libraries, or tools:** Open-source libraries, code snippets from Stack Overflow, and AI-generated code all require attribution.

**Tips for Technical Citations:**

- Include version numbers when citing software or documentation that changes between versions.
- Use retrieval dates only for sources that change frequently and lack version numbers.
- DOIs are preferred over URLs when available.
- When in doubt, consult the [APA Style website](https://apastyle.apa.org/) or Freel Library resources.

When in doubt, cite. It is always better to over-cite than to inadvertently present someone else's work as your own.

### In-Text Citations

**Single Author:**

> According to Fowler (2018), refactoring improves code maintainability without changing external behavior.

**Two Authors:**

> React's component-based architecture promotes reusability (Banks & Porcello, 2020).

**Three or More Authors:**

> Modern authentication systems should implement multi-factor authentication (Smith et al., 2021).

**Organization as Author:**

> The OWASP Foundation (2023) identifies injection attacks as a top security vulnerability.

**Direct Quote:**

> "PostgreSQL provides robust support for complex queries and data integrity" (PostgreSQL Global Development Group, 2024, para. 3).

### Reference List Examples

**Official Documentation (Web Page):**
